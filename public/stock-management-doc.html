Estructura de Datos del Stock
Cómo se almacena el stock en la base de datos

El stock no se almacena en una tabla independiente por producto. En cambio, cada Product y ProductVariant tiene un campo JSON llamado stock_channels que contiene el stock disponible desglosado por canal de venta.

Formato del campo stock_channels

// Ejemplo de stock_channels en un producto o variante
stock_channels = [
  {
    "channel":       1,               // ID del canal
    "channel_name":  "Tienda Online",  // Nombre descriptivo
    "stock_quantity": 100,            // Stock disponible actual
    "stock_alert":    10             // Umbral de alerta (envía email)
  },
  {
    "channel":       2,
    "channel_name":  "Local Comercial",
    "stock_quantity": 50,
    "stock_alert":    5
  }
]
Product
app/Models/Product.php

Campo stock_channels casteado como array. Se usa cuando el producto no tiene variantes activas.
ProductVariant
app/Models/ProductVariant.php

Campo stock_channels casteado como array. Tiene prioridad sobre el producto cuando existe y no está vacío.
ℹ️
Regla de prioridad Si el ProductVariant tiene stock_channels configurado, ese valor se usa para la validación y descuento. De lo contrario, se usa el del Product. Si ninguno tiene stock_channels, el producto se considera sin control de stock y se omite en las validaciones.

Canales de Venta
Cada canal tiene su propio stock independiente
Canal IDNombreFlujo de aprobación
1Tienda OnlineManual o por webhook de MercadoPago

StockService
app/Services/StockService.php – Servicio central de gestión de stock

Clase estática que concentra toda la lógica de validación, descuento y restauración de stock. Todos los cambios de inventario pasan por este servicio.

4.1  validateStock(Sale $sale): array

Verifica que todos los productos de una venta tengan stock suficiente en el canal correspondiente. No modifica datos.

ENTRADA: $sale con relaciones products.product y products.variant cargadas PARA CADA producto en $sale->products: │ ├─ ¿El variant tiene stock_channels? │ SÍ → Usar variant.stock_channels │ NO → Usar product.stock_channels │ ├─ ¿stock_channels está vacío? │ SÍ → SALTAR (sin control de stock) │ ├─ Buscar el canal que coincida con $sale->channel_id │ └─ ¿stock_quantity < cantidad_solicitada? SÍ → Agregar error al array: │ { product_id, product_variant_id, │ product_name, requested, available } NO → Continuar RETORNO: array de errores (vacío = OK)
✅
Formato de respuesta de error Cada objeto en el array incluye product_name (con nombre de variante si aplica), la cantidad solicitada y la disponible, para mostrar al usuario un mensaje descriptivo.

4.2  discountStock(Sale $sale): void

Descuenta el stock al momento de aprobarse una venta. Siempre se llama después de validateStock().

PARA CADA producto en $sale->products: │ ├─ Determinar fuente de stock (variant o product) │ ├─ Recorrer stock_channels por referencia: │ Si channel == $sale->channel_id: │ stock_quantity = max(0, stock_quantity - cantidad) │ (nunca baja de cero) │ ├─ Guardar el modelo (variant.save() o product.save()) │ └─ Registrar movimiento en stock_movements: quantity: -cantidad (negativo = salida) note: "Deducción por confirmación de pedido #ID" AL FINALIZAR todos los productos: └─ StockAlertService::checkAndNotify() para cada producto afectado

4.3  restoreStock(Sale $sale): void

Revierte un descuento de stock cuando una venta aprobada es cancelada.

PARA CADA producto en $sale->products: │ ├─ Determinar fuente de stock (variant o product) │ ├─ Recorrer stock_channels por referencia: │ Si channel == $sale->channel_id: │ stock_quantity = stock_quantity + cantidad │ (sin límite superior) │ ├─ Guardar el modelo │ └─ Registrar movimiento en stock_movements: quantity: +cantidad (positivo = ingreso) note: "Restauración por cancelación de pedido #ID" NOTA: NO se ejecutan alertas de stock en restauraciones.
⚠️
Diferencia importante entre descuento y restauración discountStock() dispara alertas de stock bajo. restoreStock() no las dispara, ya que un aumento de stock no requiere notificación.

4.4  recordMovement() (privado)

Registra cada cambio de stock como una entrada en la tabla stock_movements. Envuelto en try/catch para que un fallo de auditoría no interrumpa la operación principal.

ParámetroTipoDescripción
$productIdintID del producto afectado
$variantId?intID de la variante (null si no aplica)
$quantityintNegativo para salidas, positivo para ingresos
$notestringRazón legible del movimiento
$saleId?intID de la venta relacionada
$userId?intUsuario que realizó la acción

StockAlertService
app/Services/StockAlertService.php – Notificaciones de stock bajo

Servicio que evalúa si el stock de un producto cruzó el umbral de alerta y envía un email de notificación al equipo interno.

Método checkAndNotify(Product $product)

ENTRADA: Product (recargado con ->fresh()) Cargar variantes activas (sin soft-delete) ¿Tiene variantes activas? │ ├─ SÍ → Evaluar cada variant.stock_channels │ PARA CADA canal: │ SI stock_quantity <= stock_alert: │ Agregar alerta { variante, canal, stock_actual, umbral } │ └─ NO → Evaluar product.stock_channels PARA CADA canal: SI stock_quantity <= stock_alert: Agregar alerta { producto, canal, stock_actual, umbral } ¿Hay alertas? └─ SÍ → Enviar email a info@etiquecosas.com.ar via Mailable: StockAlertMail Log::info() con detalle
Cuándo se dispara
Después de cada llamada a StockService::discountStock(). Es decir, al aprobar cualquier venta.
Cuándo NO se dispara
En restoreStock() (cancelaciones) y en ajustes manuales de stock desde el panel de administración.

app/Models/StockMovement.php – Registro inmutable de movimientos

Tabla de auditoría append-only que registra cada entrada y salida de stock. No tiene updated_at para reforzar su carácter inmutable.

Flujo 1: Venta Online con MercadoPago
El camino más común para ventas de la tienda online
1
Cliente completa el checkout – POST /v1/sales
Se crea la venta con sale_status_id = 8 (Pendiente de pago). Los productos se vinculan a la venta. No se valida ni descuenta stock en este paso.
2
Frontend solicita enlace de pago – POST /v1/mercadopago/create-preference
Se crea la preferencia en MercadoPago y se retorna el init_point. No se valida ni descuenta stock en este paso.
3
Cliente realiza el pago en MercadoPago
El cliente es redirigido a la plataforma de MercadoPago. El sistema espera la notificación (webhook).
4
MercadoPago notifica el resultado – POST /mercadopago/webhook
El webhook verifica que la venta no haya sido aprobada antes (protección contra doble ejecución). Si el pago es aprobado: cambia el estado a 1, envía emails, descuenta stock, registra movimientos y dispara alertas. Genera PDFs de forma asíncrona. Si es rechazado: cambia el estado a 9, sin impacto en stock.
[Cliente] [API] [MercadoPago] [Stock] │ │ │ │ │─ POST /sales ───►│ │ │ │ │ Crea venta (status=8) │ │ │◄─ sale_id ───────│ │ │ │ │ │ │ │─ create-pref ───►│ │ │ │ │─ POST /preferences ───►│ │ │◄─ init_point ────│◄─ preference ──────────│ │ │ │ │ │ │─ paga ──────────►│(MercadoPago) │ │ │ │◄─ webhook (approved) ──│ │ │ │─ discountStock() ──────┼─────────────────────►│ │ │─ status = 1 │ │ │◄─ email conf. ───│ │ │
🔒
Protección contra doble aprobación El webhook verifica con hasBeenApproved() que la venta no tenga historial de estado 1. Esto evita que un webhook duplicado descuente el stock dos veces.

Flujo 2: Venta Local (en tienda)
SaleController@storeLocalSale – Canal 2

Las ventas locales se aprueban de forma inmediata. La validación y el descuento de stock ocurren en el mismo request de creación. Si el stock falla, la venta es eliminada.

1
Admin crea la venta – POST /sales/local
Se crea la venta directamente con sale_status_id = 1 (Aprobada) y channel_id = 2.
2
Validación de stock
StockService::validateStock($sale). Si falla, se eliminan los productos y la venta, y se retorna HTTP 422.
3
Descuento inmediato
StockService::discountStock($sale). Se descuenta el stock, se registran movimientos y se verifican alertas.
4
Registro en historial
Se crea un SaleStatusHistory para la transición al estado 1.

Actualización de venta local – PUT /sales/local/:id

1
Restauración del stock anterior
StockService::restoreStock($sale) para devolver el stock de los productos originales.
2
Reemplazo de productos
Se eliminan los SaleProduct anteriores y se crean los nuevos.
3
Descuento del nuevo stock
StockService::discountStock($sale) con los productos actualizados.

Flujo 3: Venta Mayorista (Canal 4)
SaleController@store con channel_id = 4
1
Se crea la venta en estado 8
Igual que una venta online estándar.
2
Auto-aprobación inmediata
Por ser canal 4, se llama automáticamente a approveSale() que valida stock y, si todo está bien, descuenta.
3
Si no hay stock: rollback a estado 8
Se captura InsufficientStockException, el estado vuelve a 8 y se retorna error 422. La venta queda registrada pero sin descuento de stock.

Flujo 4: Aprobación Manual por Admin
SaleController@changeStatusAdmin
1
Admin cambia el estado a 1 (Aprobada)
El sistema verifica con hasBeenApproved() que la venta no haya sido aprobada antes. Si ya fue aprobada, el request es rechazado.
2
Validación y descuento
Se llama a approveSale() que valida stock y descuenta. Si falla, lanza InsufficientStockException con el detalle.

Flujo 5: Cancelación y Restauración
Qué pasa cuando una venta aprobada se cancela
⚠️
Solo se restaura stock si la venta fue aprobada Si una venta en estado 8 o 9 se cancela, no se ejecuta ninguna restauración de stock porque nunca se descontó.
1
Admin cambia el estado a 5 (Cancelada)
El controlador verifica si la venta tiene historial de estado 1 (fue aprobada).
2
Si fue aprobada: restaura stock
StockService::restoreStock($sale) suma la cantidad de vuelta a cada canal correspondiente.
3
Registro de movimiento
Se registra en stock_movements con cantidad positiva y nota de restauración. No se disparan alertas.

InsufficientStockException y respuestas HTTP

Clase InsufficientStockException

Extiende RuntimeException. Se lanza dentro de approveSale() cuando la validación falla, y es capturada por los controllers para generar la respuesta HTTP adecuada.

Respuesta HTTP 422 – Stock insuficiente

{
  "message": "Stock insuficiente para confirmar el pedido",
  "data": [
    {
      "product_id":         42,
      "product_variant_id": 7,
      "product_name":       "Etiqueta Tejida - Color Rojo",
      "requested":          5,
      "available":          2
    }
  ]
}

Resumen y Salvaguardas
Principios de diseño del sistema de stock
AspectoImplementación
AlmacenamientoJSON stock_channels en Product y ProductVariant, por canal
ValidaciónStockService::validateStock() – antes de cualquier descuento
DescuentoStockService::discountStock() – solo al aprobar
RestauraciónStockService::restoreStock() – solo al cancelar ventas aprobadas
AlertasStockAlertService::checkAndNotify() – email cuando stock ≤ umbral
AuditoríaTabla stock_movements – append-only, cada cambio registrado
ErroresInsufficientStockException – HTTP 422 con detalle por producto

Salvaguardas implementadas

Anti doble-aprobación
Sale::hasBeenApproved() consulta el historial de estados. Si ya existe un registro de estado 1, se rechaza la operación. Previene doble descuento de stock por webhooks duplicados.
Stock nunca negativo
max(0, stock_quantity - quantity) garantiza que el stock no baje de cero, incluso si hay inconsistencias en los datos.
Validar antes de modificar
En todos los flujos, validateStock() se ejecuta antes de discountStock(). Nunca se descuenta stock sin validar primero.
Auditoría no bloqueante
recordMovement() está envuelto en try/catch. Un fallo al registrar el movimiento no interrumpe la operación de stock principal.
